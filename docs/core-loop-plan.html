The core loop

SpeechPractice is an objective, closed-loop trainer for scripted-speech clarity: it measures how intelligibly you read, finds your specific weak spots, writes you material dense in exactly those targets, and proves — with the same deterministic ruler — that you fixed them.

1What it is, what it isn't

The gen-1 core (read → transcribe → align → score → highlight) is a measurement instrument. Everything we're adding is a coaching loop wrapped around that instrument — and every new feature must be justifiable as a step in that loop.

  • The ruler is machine intelligibility. Clarity = 1 − flexible WER against Whisper. That's a proxy for human intelligibility — imperfect, but objective and comparable across months. The self-review flow is the human override for when the proxy mishears.
  • Not a phoneme-level pronunciation tutor (ASR can't see formants), not a content or delivery coach. Free Speak stays a fluency dipstick — pace, pauses, fillers — with no accuracy pretense.

2The loop

Five stages, each with exactly one job. The looseness in the current app is that stages 3→4→5 don't connect: practicing a drill doesn't reliably credit its card, and nothing ever proves a card fixed.

MeasureReading sessions produce immutable evidence: scores + per-token error events.
→
DiagnoseAnalytics proposes cards; self-review confirms or adds. A card = a hypothesis with evidence.
→
TreatAI writes drills & ladders dense in the card's targets. Ladders are the canonical treatment.
→
VerifyRe-measure on treatment material. Each card graded on its own target's performance.
→
Retire & watchMastered cards leave the queue; analytics keeps sentinel duty on the target.

↩ Lapse path: a retired target that re-errors in ordinary reading sessions re-enters at Diagnose — the card comes back with fresh evidence, not from scratch.

The two rules that keep it coherent

Rule A — Cards are entities, not views.

Analytics proposes new cards and refreshes evidence (stats, prompt, title). Reviews progress them (mastery, status, due date). Analytics never writes SRS state on an existing card. Today refresh_improvement_cards clobbers mastery after every session — this is the first thing we fix.

Rule B — AI generates, deterministic code grades.

LLMs write drill text and translate self-review notes into cards. Scoring, per-target grading, mastery, and rung gates stay pure alignment math, so a 92% clarity in March means the same as a 92% in July. No model ever assigns mastery or passes a rung.

3Verify: per-target grading is the core, not phase 2

Today every linked card gets the same session-global quality (65% score + 35% WER). That smears one number across all cards and makes mastery mean "you read well lately," not "you fixed this." Instead, each card kind gets an evidence extractor that answers, for one session: how many chances did this target get, and how many did it miss?

The quality formula

Target evidence is blended with session quality using a shrinkage weight, so two occurrences of a word don't swing mastery as hard as ten:

target_quality  = 1 − (misses / opportunities)
w               = opportunities / (opportunities + 3)     # trust grows with evidence
quality         = w · target_quality + (1 − w) · session_quality

# session_quality stays the existing blend: 65% scaled score + 35% (1 − WER)
# opportunities == 0  →  no review for that card (no evidence, no credit)
# fluency cards       →  always graded, from the session's fluency metrics

The "no evidence, no credit" rule matters: a ladder level that happens to omit a target simply doesn't touch that card, instead of granting it free mastery. Generated ladders weave every target into every level, so in practice all ladder cards get graded on every rung.

Which cards a session updates (fan-out)

When a scoring job succeeds, the set of candidate cards is the union of:

  • the explicit card on the job (quick-practice form selection),
  • if the script is a ladder step: all cards on the ladder's cards M2M (focus lists are effectively ladder-wide by design — every level weaves every target),
  • all GeneratedPracticeScript links for the script — not the current arbitrary .first().

Dedupe, then run one evidence-graded review per card. PracticeReview records the per-card quality plus its evidence (opportunities, misses) so the card's history is auditable.

4Card cleanup: drop phrase cards

Phrase cards are one-off n-grams from a single bad take — they don't name a reusable skill, they bloat the queue, and there's no honest per-session evidence extractor for them (the exact phrase rarely reappears in new material).

  • Remove the phrase block from build_card_candidates — trends stop proposing them.
  • Remove phrase from the self-review card prompt's kind list (a phrase-shaped complaint becomes a word, sound, or fluency card instead).
  • Data migration pauses existing phrase cards rather than deleting — history and reviews stay intact, they just leave the queue and ladder-generation pickers.
  • The model choice constant stays (old rows reference it); it's just no longer produced.

5Ladder gates: mastery in context

"Mastered" shouldn't mean your average went up — it means the target survived progressively harder contexts. The five ladder levels are exactly that difficulty gradient, so gating rungs on clarity is what makes Verify real.

  • Rule: rung N+1 unlocks when rung N has a session with clarity ≥ its min_clarity. Level 1 is always open. (Clarity is 1 − flexible WER, so this is equivalently a WER ceiling.)
  • Threshold: new min_clarity field on PracticeLadderStep. Generated ladders ramp 85 → 88 → 90 → 92 → 95 to match the warm-up → performance level design; plain default is 90 (the "good" clarity band).
  • State: new LadderStepProgress model — (user, step) unique, with best_clarity, attempts, passed_at, best_session_id. Upserted when a scoring job for a ladder-step script succeeds.
  • Enforcement in three places: GET for a locked level redirects to the highest unlocked rung with a message; POST rejects a locked rung's script (no URL-crafting around the gate); the template renders locked rungs disabled with the requirement stated.
  • Backfill: management command walks succeeded ScoringJobs for ladder-step scripts → legacy_session_id → session clarity, and builds progress rows — existing users keep their earned unlocks.
  • Unlock moment: the job-status payload (already polled) carries a "Level 4 unlocked" event for the results page.

7Decisions ledger

What we've settled in discussion, so the build doesn't relitigate it.

Ladder fan-out
All cards on the ladder's M2M — focus lists are ladder-wide in practice (every level weaves every target by design), so no focus-matching.
Gate metric
Clarity only (= 1 − flexible WER). No separate WER ceiling — it's the same number.
Per-target grading
In the core loop from the start (phase 1), not deferred. No evidence → no credit.
Phrase cards
Removed from generation; existing ones paused, history kept.
Backfill
Yes — historical sessions count toward rung unlocks.
Clobber fix
Prerequisite, ships first (phase 0).
AI's role
Generates material and translates self-review notes. Never grades, never assigns mastery, never passes a rung.

Still open (small)

  • Builtin/imported ladders with no linked cards: gates work (progress is per-step), card updates are simply a no-op. Acceptable, or should imported ladders get a card-linking step?
  • Shrinkage constant (k = 3) and the 85→95 ramp are starting values — tune after a week of real use.
  • Does a failed gate attempt still schedule the card (quality < 0.5 → due tomorrow)? Current SRS says yes; feels right, flagging it as intended behavior.
